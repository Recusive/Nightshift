# Claude cycle-result contract drift should not false-reject real fixes

Evaluation `#0015` started cleanly, ran baseline verification, and passed both
cycle verification commands against Phractal, but both cycles were still
rejected because the Claude JSON payload used summary/count fields such as
`fixes_committed`, `fixes_applied`, `commit_shas`, and `summary` instead of the
schema-shaped `fixes[]` / `logged_issues[]` structure that `parse_cycle_result()`
and `verify_cycle()` expect.

That drift collapses the structured result to zero fixes, so valid non-log code
commits are treated as unexpected work (`structured output implies 0`), which
keeps real evaluation runs below the acceptance threshold even when the code
changes themselves verify cleanly.

## Acceptance Criteria

- [ ] Claude-backed cycle runs either emit the documented schema shape or the
      parser/verifier normalizes the count-based payload without false
      `structured output implies 0` rejections
- [ ] Regression coverage reproduces the `fixes_committed` / `fixes_applied`
      payload shape from Evaluation `#0015`
- [ ] A reproduction matching Evaluation `#0015` no longer rejects a valid
      single-fix cycle solely because the structured payload used summary/count
      fields
- [ ] Real-run artifact fidelity is preserved: the state file or rejected-run
      summary still makes it clear what Claude reported, even when details are
      partial
