# Self-evaluation loop — score the system against real repos

The builder ships PRs but never measures whether they actually improve target codebases. The evaluation system (docs/evaluations/) was designed but never activated. Without this feedback loop, the system optimizes for shipping, not for impact.

## Design

After every N builder sessions (configurable, default 5), the daemon runs a mini-evaluation:

1. Clone the test target repo (Phractal or configurable via `.nightshift.json`)
2. Run a 2-cycle test shift: `nightshift test --agent claude --cycles 2`
3. Score the output across key dimensions: did it find real issues? did fixes pass verification? was the shift log useful?
4. Write a brief evaluation report to `docs/evaluations/NNNN.md`
5. If any dimension scores below threshold, create a task

This is different from tasks #0012/#0028 (which are one-time validations). This is a recurring health check built into the daemon loop.

## Verification

1. Run: `PYTHONPATH=. python3 -m nightshift test --agent claude --cycles 1 --dry-run` — verify the command works
2. Run one real evaluation cycle against Phractal — verify it produces a shift log and state file
3. Check: `docs/evaluations/0001.md` exists with scores
4. Check: if any dimension is below 6/10, a task was created in `docs/tasks/`

## Acceptance Criteria

- [ ] Evaluation runs automatically every N sessions (configurable)
- [ ] Clones target repo, runs test shift, scores results
- [ ] Writes evaluation report to `docs/evaluations/`
- [ ] Creates follow-up tasks for low-scoring dimensions
- [ ] Does not block the builder if evaluation fails (logs and continues)
- [ ] Configurable target repo and frequency in `.nightshift.json`
