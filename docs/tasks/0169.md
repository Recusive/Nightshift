# Fix extract_result_summary to handle Codex stream format

## Problem

`extract_result_summary` in `scripts/lib-agent.sh` only parses Claude's JSONL format:

```json
{"type": "result", "result": "PENTEST REPORT\n..."}
```

Codex `--json` emits a different format:

```json
{"type": "item.completed", "item": {"type": "agent_message", "text": "PENTEST REPORT\n..."}}
```

`PENTEST_AGENT` defaults to `$AGENT` (line 205 of `daemon.sh`). When running the
Codex daemon (`daemon.sh codex`), the pentest agent is also Codex. The current
parser finds zero `type: result` events and silently exits — `PENTEST_REPORT`
is always empty, so the builder sees:

```
<pentest_data status="success">
No structured pentest report was produced.
</pentest_data>
```

This is a **false-green**: the pentest agent runs successfully and produces a real
report, but the builder never sees it.

## Scope

Three extractors in `daemon.sh` use the identical `type: result` pattern and are equally broken
for Codex. All three must be fixed together:

1. `extract_result_summary` in `lib-agent.sh` — used for `PENTEST_REPORT` at daemon.sh:234
2. `FEATURE` inline Python at daemon.sh:384-397 — parses "Built: ..." from session result
3. `PR_URL` inline Python at daemon.sh:399-412 — parses "PR: ..." from session result

When running the Codex daemon, items 2 and 3 always write "-/-" for feature name and PR URL to
the session index, making the index unreliable for monitoring and role scoring.

## Evidence

- `scripts/lib-agent.sh`: `extract_result_summary` reads only `event.get("type") == "result"`
- `scripts/daemon.sh:384-412`: FEATURE and PR_URL extractors use the same pattern
- `scripts/format-stream.py:49-93`: Codex format confirmed as `item.completed` /
  `item.agent_message`
- Existing test `TestExtractResultSummaryHelper` (test_nightshift.py:9816) feeds only
  `{"type":"result"}` events — no Codex-format coverage

## Fix

### 1. `extract_result_summary` (lib-agent.sh)

Update the Python inline script to handle both formats:

1. Accumulate the last `agent_message` text from `item.completed` events (Codex path)
2. Prefer the `result` field from `type: result` events (Claude path) if present
3. Fall back to the last Codex `agent_message` if no Claude result found

```python
result_text = ""
codex_last_message = ""
for raw_line in ...:
    ...
    if event.get("type") == "result":
        payload = event.get("result")
        if isinstance(payload, str) and payload.strip():
            result_text = payload.strip()
    elif event.get("type") == "item.completed":
        item = event.get("item", {})
        if item.get("type") == "agent_message":
            text = item.get("text", "")
            if isinstance(text, str) and text.strip():
                codex_last_message = text.strip()
if not result_text:
    result_text = codex_last_message
```

### 2. FEATURE extractor (daemon.sh:384-397)

Apply the same dual-format logic. The `Built:` line is in the result text, so once
`extract_result_summary` is fixed, this extractor can call it or use the same pattern.
Simplest path: extract FEATURE/PR_URL from the result text returned by
`extract_result_summary` rather than re-parsing the log file independently.

### 3. PR_URL extractor (daemon.sh:399-412)

Same approach as FEATURE. Parse "PR: ..." from the result text, falling back to Codex format.
