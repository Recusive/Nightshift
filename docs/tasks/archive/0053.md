# Agent generates its own tasks across all dimensions

Right now the agent only picks tasks from the queue. It never asks itself "what should I be working on?" It waits to be told. That's not autonomous -- that's a task runner.

After every session, the agent should look at the entire system and generate tasks across every dimension it can think of. Not just "what's next in the queue" but "what would make this whole thing better?"

## The dimensions

The agent should think about ALL of these every session and create tasks when it spots opportunities:

**Meta layer / autonomous pipeline:**
- How can the daemon loop be more reliable?
- Are prompts getting stale or bloated?
- Is the task system working or causing friction?
- Are handoffs useful or just noise?
- Is cost per session trending up or down?
- Are sessions getting stuck in patterns?

**Code quality:**
- Are there modules that are too big and should be split?
- Are there functions with no tests?
- Are there type annotations that are too loose?
- Is there dead code nobody uses?
- Are there TODO comments that never became tasks?
- Are error messages helpful or cryptic?

**Repo health:**
- Is CI fast or slow? Can it be faster?
- Are dependencies up to date?
- Is the test suite comprehensive or has coverage drifted?
- Are there flaky tests?
- Is documentation accurate or has it drifted from reality?

**Architecture:**
- Are there circular dependencies forming?
- Is the module structure still clean or getting tangled?
- Are abstractions earning their keep or adding complexity?
- Is the config system growing beyond what's needed?

**Developer experience (for agents):**
- Is CLAUDE.md still accurate?
- Are learnings being applied or just written?
- Is the handoff format giving the next session what it needs?
- Can a fresh agent session get productive faster?

**Vision progress:**
- What would move the tracker percentage the most?
- Are there low-hanging fruit that would close out a section?
- Are there blocked items that could be unblocked with a different approach?
- Is the agent avoiding any area of the vision?

**Security / robustness:**
- Are there edge cases that would crash the system?
- Is input validation sufficient at boundaries?
- Could a malicious PR exploit the auto-merge flow?
- Are secrets/credentials properly excluded everywhere?

## How it works

Add a new step to the evolve prompt, after the handoff is written:

```
STEP 11 -- GENERATE WORK

Before ending the session, look at the system from every angle:
- Read the vision tracker. What's furthest behind?
- Read the last 5 session index entries. Any patterns?
- Scan the codebase. Any TODOs, hacks, or weak spots?
- Think about the meta layer. Any friction you hit this session?
- Think about what annoyed you or slowed you down.

For each issue you find, create a task in docs/tasks/ using .next-id.
Tag them appropriately (priority, environment, target).

You are not just a builder. You are the engineer who owns this system.
Act like it.
```

## Constraints

- Don't flood the queue -- max 3-5 new tasks per session
- Don't create duplicate tasks (scan existing pending tasks first)
- Don't create vague tasks ("improve code quality") -- each task must have specific acceptance criteria
- Tasks should span different dimensions, not all the same type
- Priority should be honest -- not everything is urgent

## Acceptance Criteria

- evolve.md includes a "generate work" step after handoff
- Agent creates 1-5 new tasks per session based on what it observes
- Tasks span multiple dimensions (not all code quality, not all meta)
- Tasks have specific acceptance criteria
- Agent checks for duplicates before creating
- Queue doesn't explode (max 5 per session enforced in prompt)
