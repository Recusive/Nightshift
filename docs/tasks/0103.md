# CI/CD pipeline for autonomous agent feedback — zero human intervention

Blocked for now: this is an umbrella epic spanning CI artifacts, coverage
history, post-merge remediation, scheduled Phractal evaluation, and
self-healing automation. It overlaps existing queued tasks such as `#0076`,
`#0093`, and `#0094`, so it needs decomposition into concrete deliverables
before an autonomous builder session can complete it without partial work.

## Vision

The CI/CD pipeline should be built BY the agent, FOR the agent. Right now CI is a basic pass/fail gate. The autonomous agent needs rich, structured feedback from CI so it can self-diagnose, self-fix, and maintain this codebase with zero human intervention.

## What the agent needs from CI

### 1. Structured failure reports
- When CI fails, the agent gets a raw log dump. It needs structured output:
  - Which check failed (lint, types, tests, docs validation)
  - Exact file:line references
  - Suggested fix category (import error, type mismatch, missing test, etc.)
- Output as JSON or markdown the agent can parse in the next session

### 2. Test health dashboard
- Test count tracking over time (detect regressions: "tests dropped from 891 to 850")
- Flaky test detection (same test fails intermittently across sessions)
- Coverage delta per PR (not just pass/fail — did coverage go up or down?)
- Slow test detection (tests that are dragging CI time up)

### 3. Post-merge validation
- After every merge to main, run the full `make check` AND a smoke test
- If post-merge fails, auto-create a GitHub issue with `needs-human` label if the agent can't fix it in 2 attempts
- Track consecutive main failures — circuit breaker for the repo, not just the daemon

### 4. E2E evaluation in CI
- Run `nightshift test --agent claude --cycles 1 --cycle-minutes 3` against Phractal on a schedule (nightly or every N merges)
- Store evaluation scores over time — trend line visible to the agent
- Alert if score drops below threshold (e.g., < 40/100)

### 5. Dependency and security scanning
- Automated dependency freshness checks (are we pinned to old versions?)
- Security scanning for known vulnerabilities in dependencies
- License compliance (if we ever go open source)

### 6. Agent-readable CI artifacts
- Every CI run should produce a `ci-report.json` artifact the agent can download and parse
- Fields: checks_run, checks_passed, checks_failed, failure_details[], test_count, coverage_pct, lint_warnings, type_errors, duration_seconds
- The daemon's housekeeping can pull this before each session for awareness

### 7. Self-healing CI
- If a CI check fails on a pattern the agent has seen before (e.g., ruff formatting), auto-create a fix branch
- If the fix is trivial (formatting, import order), auto-PR and merge
- If non-trivial, create a task with the structured failure report

## Implementation approach

- Build on GitHub Actions (already in use)
- The agent should design and implement the workflow files itself
- Each capability above can be a separate job in the pipeline
- Start with structured failure reports (#1) and agent-readable artifacts (#6) — these unblock everything else

## Acceptance Criteria

- [ ] CI produces structured JSON report as artifact on every run
- [ ] Agent can parse CI report in daemon housekeeping step
- [ ] Test count and coverage tracked over time
- [ ] Post-merge validation with auto-issue creation on failure
- [ ] E2E evaluation runs on schedule with score tracking
- [ ] At least one self-healing pattern implemented (e.g., auto-fix ruff formatting)
- [ ] All CI workflows documented in docs/ops/OPERATIONS.md
- [ ] Zero human intervention required for routine CI failures

## Why this matters

This is the bridge from "agent that ships PRs" to "agent that maintains a production system." Without rich CI feedback, the agent is blind after merge. With it, the agent closes the loop: build → ship → verify → learn → fix. Full autonomy.
