# Multi-agent PR review panel that scales with task complexity

Currently the builder spawns one sub-agent (code-reviewer) to review every PR. Same depth for a 3-line docs fix and a 500-line new module. Wasteful on small PRs, insufficient on complex ones.

## Design

The builder determines review depth based on what changed, then spawns the right number of sub-agents in parallel.

**Baseline (always runs, all PRs):**
1. **Code reviewer** — structure, types, registration, tests (existing agent, unchanged)

**Added for medium PRs (new modules, >100 lines changed, new dependencies):**
2. **Safety reviewer** — security, no hardcoded paths, no secrets, no force push, subprocess usage
3. **Docs reviewer** — changelog, CLAUDE.md, OPERATIONS.md, handoff, tracker consistency

**Added for complex PRs (new subsystem, >300 lines, touches 5+ files, changes daemon/prompt/meta layer):**
4. **Architecture reviewer** — dependency flow, module boundaries, does it belong here, is it over-engineered
5. (optional) **Meta reviewer** — only when PR touches scripts/, docs/prompt/, daemon config. Checks: will this break the autonomous loop?

## How the builder decides

Simple heuristic in the prompt — no code needed:

```
Count files changed and lines changed in `gh pr diff --stat`.

- Docs-only (.md only):        1 agent  (code reviewer, fast path)
- Small (<100 lines, <3 files): 1 agent  (code reviewer)
- Medium (100-300 lines or new module): 3 agents (code + safety + docs)
- Complex (>300 lines or 5+ files or touches meta layer): 4-5 agents
```

The builder spawns them in parallel, collects results, and only merges if ALL pass.

## Live integration test (not unit tests)

After all reviewers pass but BEFORE merging, the builder runs a real integration test against the actual system. Not pytest mocks — a real end-to-end run that proves the code works.

**What this means:** If the PR touches the orchestrator, run a real `nightshift harden` dry-run against a test repo. If it touches the daemon scripts, run the daemon for one session and check the exit code. If it touches cost tracking, parse a real session log and verify the numbers. If it touches the task system, create a task, read it back, archive it.

The builder should write a small throwaway test script (or inline shell commands) that exercises the exact code path the PR changed — with real files, real git, real CLI invocations. Not assertions in a test file. Actual behavior.

```
# Examples of what "real test" means:

# PR touches nightshift/scoring.py:
python3 -c "from nightshift.scoring import score_diff; print(score_diff('...'))"
# Did it return a sane number? Did it crash?

# PR touches daemon.sh:
bash scripts/daemon.sh claude 0 1  # run 1 session, 0 pause
# Did it exit 0? Did it produce a log?

# PR touches cost tracking:
python3 -c "from nightshift.costs import parse_session_tokens; ..."
# Feed it a real log, check the output

# PR touches task system:
echo "50" > docs/tasks/.next-id && cat docs/tasks/.next-id
# Did the round-trip work?
```

This is the last gate before merge. Reviewers check the code looks right. The integration test checks it actually works.

## Acceptance Criteria

- 4-5 agent definition files in `.claude/agents/`
- Each agent has a focused scope (no overlap)
- `evolve.md` step 5 includes the complexity heuristic
- Small PRs still get 1 reviewer (no regression in speed)
- Complex PRs get 3-5 parallel reviewers
- Builder collects all results — ALL must pass to merge
- Review notes from ANY agent become follow-up tasks
