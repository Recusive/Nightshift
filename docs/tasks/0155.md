# Restore abort-on-revert-failure at both origin guard sites in daemon.sh

## Problem

Handoff #0074 documented that PR #142 added `notify_human` + `break` on exit
code 2 at both origin-guard sites in `scripts/daemon.sh`. However, PR #143
(a68ba4c) overwrote these. The current code at both sites only logs a message
when `origin_rc=2` and then falls through:

**Pentest preflight (line 236-238):** echoes a message, then falls through to
`cleanup_prompt_snapshots` + `reset_repo_state` which pulls the TAMPERED
origin/main. No abort, no notification.

**Post-builder (line 316-318):** sets PROMPT_TAMPERED string but no
`ORIGIN_REVERT_FAILED=1`, no `notify_human`, no security abort block.

Without these, a failed revert means the daemon silently continues using
attacker-controlled content from origin/main.

**Exec-chain escalation (identified by pentest session 0076):** When exit
code 2 falls through, `reset_repo_state` pulls the still-tampered origin/main.
On the next loop iteration, the self-restart check (`md5` hash comparison at
line 162) detects that `daemon.sh` changed and runs
`exec bash "$SCRIPT_DIR/daemon.sh"` -- executing the attacker's modified
daemon script with full shell privileges. A failed revert thus escalates from
"tampered prompt files" to "arbitrary code execution via the daemon process."

## Fix

Restore the abort logic from PR #142:

1. Pentest preflight exit-code-2 block: add `notify_human` + `break` before
   `reset_repo_state`
2. Post-builder exit-code-2 block: set `ORIGIN_REVERT_FAILED=1`, add
   `notify_human` + security abort with `break` after session index logging

## Acceptance Criteria

- [ ] Pentest preflight: exit code 2 triggers `notify_human` and `break` (daemon stops)
- [ ] Post-builder: exit code 2 sets `ORIGIN_REVERT_FAILED=1`, calls `notify_human`, and `break`s
- [ ] Neither path falls through to `reset_repo_state` or continues the loop after a failed revert
- [ ] `bash -n scripts/daemon.sh` passes
