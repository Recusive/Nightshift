# Fix origin guard blind spot: content-scan existing eval/autonomy files

`check_origin_integrity` had a blind spot: it checked PROMPT_GUARD_FILES (named
explicit files) for content changes and PROMPT_GUARD_DIRS for NEW files only.
An agent could push a modified EXISTING eval or autonomy report directly to
origin/main (non-merge commit) without any detection, because:

- The file is not in PROMPT_GUARD_FILES
- It is not a new file, so the new-file detection in PROMPT_GUARD_DIRS misses it

Attack path:
1. Agent pushes modified `docs/evaluations/0015.md` directly to origin/main
2. check_origin_integrity runs -- no PROMPT_GUARD_FILE matches, no new file found
3. pick-role.py reads the falsified eval score (e.g. 99/100)
4. EVAL SCORE GATE is silently defeated, BUILD unblocked

## Fix (applied in PR for this session)

Added `PROMPT_GUARD_CONTENT_DIRS` array listing `docs/evaluations` and
`docs/autonomy`. The following two functions were updated:

- `save_prompt_snapshots`: saves per-file content copies of all existing
  `[0-9]*.md` files in each `PROMPT_GUARD_CONTENT_DIRS` dir at cycle start.
- `check_origin_integrity`: new Loop 3 iterates over saved content snapshots,
  compares each against current `origin/main` content, applies the same
  non-merge-commit filter used in Loop 1 (PROMPT_GUARD_FILES). If a direct-push
  modified an existing score-gate file, it is flagged and added to
  `files_to_restore` for auto-revert.

## Acceptance criteria (all met)

- Modifying an existing `docs/evaluations/*.md` directly on origin/main via
  a non-merge commit is detected by `check_origin_integrity`.
- Legitimate PR merges that update eval/autonomy files are NOT flagged
  (non-merge-commit filter applies).
- `make check` passes with 1136 tests.
