# Build the Healer — meta-layer observer inside the daemon loop

A lightweight agent call that runs inside daemon.sh between sessions, right after housekeeping and before the builder session starts. Think of it as the human watching the terminal — the one who notices "wait, it skipped that task again" or "this is going to break in two sessions" or "why is it spending so much time on cleanup tasks when Loop 2 hasn't moved in a week?"

It's not a bug fixer. It's the human eye on the system.

## How it thinks

The healer looks at the system the way you would if you checked in after leaving it running overnight:

- **What just happened?** Read the last session log, handoff, exit code. Did it build something useful or spin its wheels?
- **What's the trend?** Read the session index. Are sessions getting slower? More expensive? Is the same task showing up in handoffs over and over without getting done?
- **What's being avoided?** Compare the task queue against what's actually getting built. Are hard tasks aging? Is the agent cherry-picking easy wins?
- **What's about to break?** Look ahead. Is the task queue running dry? Are prompts getting bloated? Is a dependency about to cause a chain of failures?
- **Is the system getting better or worse?** Compare this week's sessions to last week's. More merged PRs? Fewer failures? Lower cost per feature? Or is it regressing?

## What it observes

- Last session log (exit code, duration, cost, errors, what it actually built)
- Last handoff (did the agent skip tasks? did it leave good context for the next session?)
- Session index (patterns across sessions: repeated failures, cost trends, velocity)
- Task queue (stale tasks, tasks skipped 3+ times, aging pending tasks, queue health)
- Prompt files (bloat, drift, conflicting instructions)
- Vision tracker (is progress actually moving? or is the agent doing work that doesn't advance the vision?)
- Learnings folder (is the agent writing useful learnings? or repeating the same mistakes?)

## What it does

- Thinks about what it observed — like a human would, connecting dots across sessions
- If it spots a pattern, trend, or upcoming problem in the meta layer, creates a task with clear context on WHY this matters
- Not just "handoff has wrong test count" but "the builder has shipped 3 sessions without updating the tracker — the numbers are drifting and future sessions will start from wrong baselines"
- Writes its observations to `docs/healer/log.md` — an ongoing journal of system health
- That's it. Observe, think, create tasks. The builder does the fixing.

## What it does NOT do

- Does not touch nightshift/*.py or tests/
- Does not build features
- Does not create PRs or branches
- Does not modify prompt files directly (creates tasks for the builder to do it)
- Does not run make check or tests
- Does not create tasks for repo features — only for the meta layer (scripts/, docs/prompt/, docs/tasks/, docs/ops/, daemon config)

## Implementation

- A single agent call in daemon.sh, between housekeeping and the builder session
- Its own prompt file: `docs/prompt/healer.md` — tells it to think like a human operator watching the system
- Its own docs folder: `docs/healer/` with `log.md` (journal of observations)
- Low max-turns (10-20) — it should be thoughtful but fast, not expensive
- Creates tasks using .next-id like anything else

## Acceptance Criteria

- `docs/prompt/healer.md` exists with the healer prompt
- `docs/healer/` directory exists
- `daemon.sh` calls the healer between housekeeping and builder session
- Healer reads last session log, handoff, session index, and vision tracker
- Healer identifies patterns and trends, not just point failures
- Healer creates tasks in docs/tasks/ using .next-id when it finds issues
- Healer writes observations to docs/healer/log.md
- Healer runs in under 2 minutes (low max-turns)
- Builder can pick up healer-created tasks normally
