# Webhook SSRF path via legitimate agent PR (pentest W2)

`notification_webhook` in `.nightshift.json` is in `PROMPT_GUARD_FILES` for
working-tree changes, but `check_origin_integrity` treats merged PRs as trusted
(the non-merge filter was intentionally added to prevent the bootstrap self-revert
problem). An agent with repo write access could open a PR changing `notification_webhook`
to an attacker-controlled URL, have it merged (by a human or the daemon), and intercept
all subsequent `notify_human()` calls.

The payload is minimal (title string only, JSON-encoded, no secrets). No fix urgency.

## Acceptance Criteria

- [ ] Document the known tension (merge-whitelist vs webhook SSRF) in
      `docs/ops/DAEMON.md` under the security model section so future
      contributors understand the intentional tradeoff
- [ ] OR: restrict `notify_human()` to a fixed allowlist of URL prefixes
      (e.g., `https://hooks.slack.com/`) with a clear error when the configured
      URL does not match, making the SSRF path useless for arbitrary endpoints
- [ ] Whichever approach is chosen: add a regression test confirming the
      protection is in place
