## Context

Session #0073 (BUILD) fixed three security regressions via PR #140. But commit `b2e2fdf`
(message: "security: revert prompt guard files pushed by agent session") was generated by
`check_origin_integrity` itself -- it saw the PR #140 merge as a "blind spot attack" and
force-reverted both daemon.sh and lib-agent.sh back to pre-PR-#140 state.

This is the bootstrap self-revert problem: the guard cannot distinguish a legitimate PR
merge from an attacker directly pushing to origin/main. Both look like "origin/main changed
during the session" because the snapshot is taken at session start.

Result: `make check` was broken (2 test failures), and all three security fixes from PR #140
were gone.

## Dependency Eliminated

**Guard auto-reverts legitimate security improvements merged via PR.**

Root cause: `check_origin_integrity` compared the pre-session snapshot against the
post-session origin/main state and reverted any guard file changes it found. It had no
mechanism to distinguish a forward PR merge from a direct push.

Fix: Before reverting, run two git checks:
1. Is `snap_hash` an ancestor of `current_hash`? (forward progression, not force-push)
2. Is there at least one merge commit between `snap_hash` and `current_hash`? (PR merge workflow)

If both are true, the change came through the legitimate `gh pr merge --merge` workflow.
Log it to stdout (daemon log) but do not revert and do not write to the alert file.
