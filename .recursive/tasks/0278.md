# Harden wrap_repo_instructions() against direct raw-input calls

## Problem

PR #272 neutralized prompt-closing sentinels inside `read_repo_instructions()`, which fixes the current call path. However, `wrap_repo_instructions()` still assumes its input is already sanitized. If a future caller passes raw instruction content directly into the wrapper, the same `--- end {name} ---` or `END OF UNTRUSTED REPOSITORY INSTRUCTIONS` strings could bypass the new guard.

## Fix

Move the delimiter neutralization to the wrapper boundary or otherwise enforce an explicit contract so `wrap_repo_instructions()` cannot be used unsafely with raw instruction text. Add regression coverage for direct wrapper invocation with unsanitized sentinel strings.

## Acceptance Criteria

- [ ] `wrap_repo_instructions()` neutralizes or rejects raw content containing prompt-closing sentinel strings
- [ ] A regression test covers direct wrapper invocation with unsanitized delimiter content
- [ ] Existing `read_repo_instructions()` behavior remains intact
- [ ] `make check` passes
