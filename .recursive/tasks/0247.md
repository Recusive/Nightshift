# Investigate count-only payload regression: Cycle 1 fixes[] empty in state file (eval #0017)

## Problem

Eval #0017 (2026-04-09) shows that Cycle 1's state entry has `cycles[0].fixes = []` despite the cycle committing a real fix (`1dab477`, auth.py security fix). The state counter shows `fixes=3` but only 1 structured fix object appears in the cycles array (Cycle 2).

This is the same `count-only payload` failure mode that PR #167 (task #0139) addressed. It either regressed or was not fully covering all response schemas.

Evidence:
- Cycle 1 fix commit: `1dab477` (Security fix, apps/api/app/api/v1/auth.py)
- Cycle 1 JSON response: used `fixes_committed: 1` count field, not a `fixes: [...]` list
- `cycles[0].fixes = []` in state.json
- `counters.fixes = 3` (inflated -- appears to count commits, not structured fixes)

## What To Do

1. Read `owl/cycle.py` and `owl/eval_runner.py` to find where `parse_cycle_result()` handles count vs list schemas.
2. Compare the Cycle 1 JSON response schema (has `fixes_committed` field) vs what the parser expects.
3. Fix `parse_cycle_result()` to correctly capture fix metadata when the agent uses the `fixes_committed` count field.
4. Add a regression test with a count-only JSON response to prevent reversion.
5. Run `make check` to verify.

## Acceptance Criteria

- `parse_cycle_result()` correctly populates `cycle.fixes` when agent response uses `fixes_committed` count field
- State file counter `fixes` matches the number of structured fix objects, not commit count
- Regression test added for count-only payload shape
- `make check` passes
