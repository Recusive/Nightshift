# Guard pick_session_role() against stderr-after-print corrupting the role name

## Context

Pentest-2026-04-06 (watch item): `pick_session_role()` in daemon.sh merges stdout
and stderr via `2>&1` (daemon.sh:83) and extracts the role name via `tail -1` (line 85).
This is currently correct because Python's stdout is block-buffered and flushes before
the process exits. However, if any code path in `scripts/pick-role.py` writes to stderr
AFTER the final `print(winner)` call (e.g., an atexit handler, an uncaught exception in
cleanup, or a future logging addition), `tail -1` would return the last stderr line instead
of the role name, causing the `case` match to fall through to the `build` default silently.

Current behavior is fail-safe (defaults to build), but it is a silent logic error.

## Fix

Option A: Separate stdout and stderr at the call site:
```bash
ROLE_OUTPUT=$(python3 scripts/pick-role.py 2>"$SOME_LOG")
```

Option B: Add `flush=True` to the final print in pick-role.py and add a clear sentinel:
```python
print(winner, flush=True)
```
...and document that all stderr must be emitted before the winner print.

Option C: Extract the role by looking for lines matching a known set of valid roles
rather than blindly taking `tail -1`.

Any option is acceptable; document the chosen approach in pick-role.py.

## Acceptance Criteria

- [ ] `pick_session_role()` extracts the role name robustly, either by separating
      streams or by validating that the extracted string is a known-valid role name
- [ ] If an invalid or empty role is extracted, the function falls back to `build`
      AND logs a warning (currently it silently falls through without any warning)
- [ ] Existing role selection behavior is unchanged for normal operation
