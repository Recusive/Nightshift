# Security: validate verify_command before shell execution

## Context

Pentest report #0109 (CONFIRMED-1) found that `verify_command` from `.nightshift.json`
is passed directly to `bash -lc` with no sanitisation. An attacker who controls a
target repo's `.nightshift.json` can inject arbitrary shell commands that execute
with the host user's privileges during baseline verification and post-cycle
verification.

The vulnerable call site is:
- `nightshift/core/shell.py:139` -- `run_shell_string(command, ...)` passes `command`
  verbatim to `["bash", "-lc", command]`
- Called from `nightshift/owl/cycle.py:790` (baseline) and `:935` (post-cycle verify)

The same pattern applies to commands inferred from `package.json` scripts
(`raven/integrator.py`, `raven/e2e.py`) but those are lower risk since they use
known script keys.

## Acceptance Criteria

- [ ] `verify_command` (from `.nightshift.json` or inferred) is validated against an
  allowlist of safe command prefixes before being passed to `run_shell_string`, OR
  the command is split with `shlex.split()` and run as a list (no shell expansion)
- [ ] A config-supplied `verify_command` containing shell metacharacters (`;`, `&&`,
  `|`, backticks, `$()`) that do not appear in the allowlist is rejected with a
  clear `NightshiftError` at config-load time
- [ ] Existing tests pass (`make check`)
- [ ] New tests cover: (a) safe commands are accepted, (b) injection attempts are
  rejected, (c) inferred commands (npm test, cargo test, etc.) continue to work

## Notes

Allowlist approach is simpler and more auditable than shlex.split. Suggested
safe prefixes: `npm`, `pnpm`, `yarn`, `bun`, `python3 -m`, `cargo`, `go test`,
`make`, `bash nightshift/scripts/`. Reject anything else unless it starts with
a known-safe binary.

If the allowlist approach is chosen, also validate commands inferred by
`infer_verify_command` and `infer_test_command` (they should already be allowlist-
compliant since they are hardcoded patterns, but add a runtime assertion).
