# Guard against gh --notes @ file expansion in release.py (C-4)

## Problem

`_create_github_release()` in `nightshift/infra/release.py` passes the full changelog content as the `--notes` argument to `gh release create`:

```python
run_capture(["gh", "release", "create", tag, "--title", tag, "--notes", changelog_content], ...)
```

Some versions of the `gh` CLI treat a `--notes` value that begins with `@` as a filename and read the file at that path instead of using the literal string. If the changelog file begins with `@/etc/passwd` (or any other sensitive path), that file's contents are uploaded as the GitHub release body, leaking host filesystem data to the public release page.

Although the changelog files are written by the autonomous agent and are generally controlled content, a prompt injection attack (see C-3) or a compromised agent cycle could write a malicious changelog file. The release pipeline would then inadvertently exfiltrate the referenced file contents via a public GitHub release.

## Fix

Two options (either is acceptable; option A is preferred):

**Option A: Write changelog to a tempfile, use --notes-file**

```python
import tempfile, os
with tempfile.NamedTemporaryFile(mode='w', suffix='.md', delete=False, encoding='utf-8') as f:
    f.write(changelog_content)
    notes_file = f.name
try:
    url = run_capture(["gh", "release", "create", tag, "--title", tag, "--notes-file", notes_file], ...)
finally:
    os.unlink(notes_file)
```

`--notes-file` reads from the given path; the `@` interpretation only applies to `--notes` in affected gh versions.

**Option B: Strip or escape leading @ character**

If the changelog content starts with `@`, prepend a space or newline so it is not interpreted as a filename reference. Simple but less robust.

## Acceptance Criteria

- [ ] `_create_github_release()` does not pass user-controlled content directly as `--notes` string
- [ ] Preferred fix: write content to a tempfile and use `--notes-file`, cleaning up the tempfile in a finally block
