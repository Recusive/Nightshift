# Fix ParseError.module losing subpackage context in module_map.py

## Problem

In `nightshift/infra/module_map.py`, the `ParseError` dataclass uses `path.name` (bare filename like `errors.py`) even for subpackage files. If parse errors occur in both `core/errors.py` and a hypothetical `owl/errors.py`, they would both report as `errors.py`, losing context.

## What To Do

1. In `_parse_modules`, use `path.relative_to(package_dir)` for the `ParseError.module` field when `package_dir` is provided
2. Fallback to `path.name` when `package_dir` is None
3. Add a test for parse error reporting with subpackage paths

## Acceptance Criteria

- ParseError.module shows relative path (e.g. `core/errors.py`) for subpackage files
- Existing top-level ParseError behavior unchanged
- make check passes
