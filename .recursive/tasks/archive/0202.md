# Fix security-to-.recursive/ gap: no operator path for pentest findings targeting framework code

## Problem

The friction log (2026-04-07) documents a confirmed gap: pentest findings that
target `.recursive/` code (daemon.sh, lib-agent.sh) have no standard fix path.

Current state:
- security-check: read-only, cannot fix anything
- build: cannot touch .recursive/ framework files
- evolve: requires 3+ friction entries, reads friction log only
- audit-agent: reviews only, does not fix security issues

This forces operators to either violate zone boundaries (as happened in session
#0106 -- build overrode to security-check and violated the target-operator rule)
or leave confirmed exploits unfixed indefinitely.

## Root Cause

The evolve operator's activation condition (3+ friction entries) is disconnected
from the security urgency signal. A pentest finding in .recursive/ is a security
event, not a friction event.

## Proposed Fix

Two options:
1. Allow `evolve` to activate when there are pentest tasks tagged `source: pentest`
   that target framework files (regardless of friction entry count).
2. Add a `security-evolve` mode or allow `build` to modify .recursive/ for tasks
   explicitly tagged `source: pentest` and `target: recursive`.

The evolve agent already has permission to modify Tier 1 engine files. Option 1
is simpler: just relax the activation condition in pick-role.py.

## Acceptance Criteria
- [ ] A confirmed pentest finding targeting .recursive/ code has a documented fix path
- [ ] The fix path does not require violating zone boundaries
- [ ] The evolve agent (or equivalent) activates when pentest tasks target framework code
- [ ] The brain.md documents when to delegate security-framework fixes to evolve
