# Security: sanitise GitHub issue titles in sync_github_tasks output

Already done. `.recursive/engine/lib-agent.sh` line ~848 already contains `title.replace('\n', ' ').replace('\r', ' ').replace(':', '-')` in the Python block of `sync_github_tasks`, sanitizing titles before the IFS=: shell loop. Confirmed during oversee triage 2026-04-09. Note: task #0164 tracks the distinct YAML injection via issue *bodies*, which is still pending.

## Context

Pentest report #0109 (CONFIRMED-2) found that `sync_github_tasks` in
`.recursive/engine/lib-agent.sh` prints task data as `task_num:issue_num:title`
and parses it back with `while IFS=: read -r task_num issue_num title`. GitHub
issue titles are attacker-controlled. A title containing a newline causes an
extra line to be injected into the shell loop, which can trigger unintended
`gh issue comment` and `gh issue close` calls on arbitrary issue numbers.

## Acceptance Criteria

- [ ] The Python block in `sync_github_tasks` sanitises the `title` field before
  printing: strip or replace embedded newlines (`\n`, `\r`) and colons (`:`)
  so the output format `task_num:issue_num:title` is safe for IFS=: parsing
- [ ] Alternatively, output as JSON lines (one JSON object per line) and parse
  with Python in the shell loop instead of using IFS splitting
- [ ] Manual test: creating a GitHub issue with a title containing a newline
  does not cause unintended gh API calls

## Notes

Minimal fix is one line in the Python block:
```python
title = issue.get('title', 'Untitled').replace('\n', ' ').replace('\r', ' ').replace(':', '-')
```
Add this before the `print(f'{task_num}:{issue_num}:{title}')` line.
