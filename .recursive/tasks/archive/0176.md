# Fix autonomy score first-match bug in read_latest_autonomy_score()

## Context

`read_latest_autonomy_score()` in `scripts/pick-role.py:45-57` uses
`re.search()` which returns the FIRST match of `TOTAL:\s*(\d+)/100` in
the autonomy report file.

ACHIEVE sessions write autonomy files with two TOTAL: sections: a baseline
score (earlier in the file) and an updated score (later in the file). For
example, `docs/autonomy/2026-04-06c.md` has:
- Line 34: `TOTAL: 76/100` (baseline, before the session's fix)
- Line 93: `TOTAL: 81/100` (updated, after the fix)

`re.search()` returns 76, not 81. This causes pick-role.py to under-read
the autonomy score by up to 14 points, over-scheduling ACHIEVE.

If a baseline ever falls below 70 (as in prior sessions that showed 67),
the over-boost is +50 instead of the correct +20 — a 30-point inflation
that displaces BUILD cycles.

Confirmed in pre-build pentest scan and noted in handoff #0092 as a "small
fix opportunity." No task existed until now.

## Fix

In `read_latest_autonomy_score()` at `scripts/pick-role.py:52`, replace:

```python
match = re.search(r"TOTAL:\s*(\d+)\s*/\s*100", text)
if match:
    return int(match.group(1))
```

with:

```python
matches = re.findall(r"TOTAL:\s*(\d+)\s*/\s*100", text)
if matches:
    return int(matches[-1])
```

This returns the LAST match — the updated score — rather than the first.
