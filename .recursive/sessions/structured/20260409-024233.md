$ gh pr create --title "fix: count-only payload inflates state counter in append_cycle_state (#0247)" --body "$(cat <<'EOF'
## Problem

Eval #0017 showed `counters.fixes = 3` despite only 1 real fix commit. The agent used a count-only payload (`fixes_committed: 1`) and the parser stored `fixes_count_only = 1` on the `CycleResult`. However, `append_cycle_state()` in `core/state.py` never checked `fixes_count_only` -- it fell through to the commit-count fallback, which counted all commits (fix + shift-log), inflating the counter.

Additionally, `cycles[0].fixes = []` in state.json because no structured fix objects exist when the agent uses count-only payloads (this is expected -- we cannot synthesize fix objects from a count).

## Root Cause

`append_cycle_state()` checked `fixes_count_only` only indirectly through the verifier functions (`expected_fix_commits`, `allowed_total_cycle_commits`) but never used it to compute `inferred_fix_count` for the state counter update.

The fallback path:
```python
if not fixes and not logged_issues and verification["commits"] and not state["log_only_mode"]:
    inferred_fix_count = len(verification["commits"])  # wrong: counted ALL commits
```
...fired when it should not have (when `fixes_count_only` was already set).

## Fix

`nightshift/core/state.py` -- `append_cycle_state()`: when `fixes` list is empty, check `cycle_result.fixes_count_only` first. Only fall back to commit count when that field is absent or zero.

## Tests

Three new regression tests added to `TestAppendCycleState`:
- `test_count_only_payload_uses_fixes_count_not_commit_count` -- primary regression: 1 fix, 2 commits -> counter = 1
- `test_count_only_payload_does_not_trigger_empty_cycle` -- count-only cycle with commits must not increment empty_cycles
- `test_count_only_payload_with_zero_count_falls_back_to_commits` -- `fixes_count_only=0` (falsy) still uses commit fallback

## Verification

`make check` passes: 1159 tests, ruff, mypy, dry-runs all green.
EOF
)"
```
